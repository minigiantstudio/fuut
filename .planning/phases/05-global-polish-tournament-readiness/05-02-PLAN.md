<task type="auto">
  <name>Task 2: vite-plugin-pwa with Workbox runtime caching</name>
  <files>apps/web/vite.config.ts, apps/web/src/main.tsx</files>
  <action>
    1. `bun add -d vite-plugin-pwa workbox-window`.
    2. Update `apps/web/vite.config.ts`:
       - Import `VitePWA` from `vite-plugin-pwa`.
       - Add to `plugins`:
         ```ts
         VitePWA({
           registerType: 'autoUpdate',
           injectRegister: false, // we'll register manually so we can wire the update toast
           includeAssets: ['favicon.ico', 'icon-192.png', 'icon-512.png', 'icon-maskable.png'],
           manifest: false, // we ship our own manifest.webmanifest (Task 1)
           workbox: {
             globPatterns: ['**/*.{js,css,html,svg,png,webmanifest}'],
             runtimeCaching: [
               // Supabase REST + RPC — NetworkFirst, fall back to cache only as a last resort
               {
                 urlPattern: ({ url }) => url.pathname.startsWith('/rest/v1/') || url.pathname.startsWith('/auth/v1/'),
                 handler: 'NetworkFirst',
                 options: { cacheName: 'supabase-api', networkTimeoutSeconds: 5, expiration: { maxEntries: 50, maxAgeSeconds: 60 * 60 } },
               },
               // Supabase Realtime — DO NOT cache (WebSocket / SSE)
               {
                 urlPattern: ({ url }) => url.pathname.startsWith('/realtime/v1/'),
                 handler: 'NetworkOnly',
               },
               // Google Fonts and other CDN assets — StaleWhileRevalidate
               {
                 urlPattern: ({ url }) => url.origin === 'https://fonts.gstatic.com' || url.origin === 'https://fonts.googleapis.com',
                 handler: 'StaleWhileRevalidate',
                 options: { cacheName: 'fonts' },
               },
             ],
             // Skip waiting + clients claim so the new SW takes over the page on update,
             // paired with the autoUpdate toast in Task 3.
             skipWaiting: true,
             clientsClaim: true,
           },
         })
         ```
    3. Wire the SW registration in `apps/web/src/main.tsx`:
       ```ts
       import { registerSW } from 'virtual:pwa-register';
       const updateSW = registerSW({
         onNeedRefresh() { /* see Task 3 toast */ },
         onOfflineReady() { /* optional: log "ready offline" */ },
       });
       ```
       Export `updateSW` via a tiny module (e.g. `apps/web/src/lib/pwa.ts`) so
       the update prompt component can call it.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Service Worker → origin | SW serves cached assets for the app's own origin. Cross-origin (Supabase) responses go through Workbox routes with explicit handlers. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-03 | Tampering | Service worker cache | mitigate | Use Workbox defaults for SW versioning (each build gets a unique revision). `registerType: 'autoUpdate'` ensures stale SWs are replaced; `skipWaiting + clientsClaim` enforce the new version. |
| T-05-04 | Denial of Service | Stale data | mitigate | `NetworkFirst` with 5s `networkTimeoutSeconds` for Supabase REST means slow networks fall back to cache rather than spinning forever — but real-time data (leaderboard_snapshots) is `NetworkOnly`, so users never see stale rankings as authoritative. |
| T-05-05 | Information Disclosure | Cached API responses | mitigate | Cache TTL = 1 hour for Supabase REST; sensitive data (predictions, auth) re-fetches on next visit. The supabase-js client also clears its own cache on auth state change. |
</threat_model>
