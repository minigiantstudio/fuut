  <action>
    Create `.github/workflows/ci.yml`:
    1. Triggers: `pull_request` (any branch) and `push` to `main`.
    2. Single `ci` job on `ubuntu-latest`:
       - Checkout, `oven-sh/setup-bun@v2` with `bun-version: 1.x`.
       - `bun install --frozen-lockfile`.
       - Type-check sequentially across the 3 workspaces:
         `(cd apps/web && bunx tsc --noEmit)`, then api, then `packages/types`.
       - `cd apps/web && bun run lint`.
       - Playwright: `cd apps/web && bunx playwright install --with-deps chromium`
         then `bunx playwright test`. Webserver auto-boots via `playwright.config.ts`.
       - Supabase local must run for Playwright. Use the `supabase-action/setup-cli`
         + `supabase start` step. Cache the supabase Docker images via `actions/cache`
         keyed on `supabase/config.toml`.
    3. Add `concurrency: { group: "ci-${{ github.ref }}", cancel-in-progress: true }`
       so superseded pushes don't double-spend runner minutes.
    4. Document the known pre-existing Playwright failure (`auth.spec.ts:19`) as a
       continue-on-error step OR temporarily mark the test `.skip` with a TODO
       referencing PR #29 — pick the path that keeps CI green without hiding real
       regressions.
  </action>
  <verify>
    <automated>Open a draft PR with a trivial whitespace change. Confirm the workflow runs, all steps pass except the known-failing test (handled per the choice above), and the run completes under 8 min.</automated>
  </verify>

  <name>Task 2: Migration-check workflow</name>
  <files>.github/workflows/migration-check.yml</files>
  <action>
    Create `.github/workflows/migration-check.yml`:
    1. Triggers: `pull_request` with `paths: ["supabase/migrations/**", "supabase/seed.sql", "supabase/config.toml"]`.
    2. `migrations` job on `ubuntu-latest`:
       - Checkout, install supabase CLI (`supabase-action/setup-cli@v1` or curl install).
       - `supabase start` (brings up ephemeral local stack — Postgres in Docker on the runner).
       - `supabase db reset` — applies every migration + seed against the fresh DB.
       - Sanity: `psql ... -c "SELECT count(*) FROM supabase_migrations.schema_migrations"`
         to assert all migrations registered; expect the current count
         (currently 10 — keep this assertion data-driven by counting
         `supabase/migrations/*.sql` files).
       - On failure: post the last 50 lines of the supabase CLI output via
         `gh pr comment` so the reviewer sees what broke without digging in logs.
    3. Independent of `ci.yml` so a migration-only PR doesn't pay the Playwright cost.
  </action>
  <verify>
    <automated>Open a draft PR adding a deliberately-broken migration (e.g., `SELECT nonexistent_column FROM matches;`). Confirm the workflow fails and posts the error excerpt.</automated>
  </verify>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions → repo | Workflows run with `GITHUB_TOKEN` scoped per-job. No external secrets needed (no remote DB access). |
| PR fork → secrets | Forks cannot access secrets via `pull_request` (GH default) — both workflows must work without secrets. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | CI workflow | mitigate | Pin all third-party Actions by SHA, not tag. Limit `permissions:` to `contents: read` (raise to `pull-requests: write` only on the migration-check error-comment step). |
| T-05-02 | Information Disclosure | CI logs | mitigate | No remote credentials in either workflow — both use ephemeral local Postgres. Mask any tokens via `::add-mask::` if added later. |
</threat_model>
