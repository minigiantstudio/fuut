## Summary

Phase 2 has three distinct work streams that must be sequenced carefully. The SessionContext refactor (D-08) is the foundational change — it unblocks multi-league support and the TopBar switcher, but it also touches every component that currently reads `session.leagueId`. This refactor must land first and be verified before other plans run.

League creation (D-01 through D-04) is the most novel work: a new Supabase RPC, an onboarding fork, and a confirmation screen. It can be planned as a self-contained plan once the schema migration and RPC are confirmed. The join path for already-authenticated users (a discretion item) must be resolved here since `Join.tsx` currently redirects authenticated users to `/` — this will break the multi-league use case.

The prediction countdown (D-09) and invite code regeneration (D-05) are isolated enhancements to existing components with no cross-cutting dependencies. They can run in parallel after the SessionContext refactor lands.

**Primary recommendation:** Three plans in dependency order: (1) SessionContext + multi-league refactor, (2) league creation flow + RPC, (3) countdown UX + regenerate code button.

### Pattern 1: SessionContext Multi-League Refactor

**What:** Replace the single `.maybeSingle()` membership query with a `.select("*, leagues(id, name)")` array fetch. Add `activeLeagueId` state, a `setActiveLeague(id)` action, and a derived `activeLeague` computed from the array. Preserve the existing `Session` interface shape for the active league so all downstream consumers break minimally.

**When to use:** This pattern applies whenever a user has a 1-to-many relationship with a contextual resource (a user belongs to many leagues; the app operates in one at a time).

**Blast radius of the change:** [VERIFIED: codebase grep]
- `SessionContext.tsx` — the refactor site
- `Index.tsx` — reads `session.leagueId`, `session.leagueName`, `session.role` — all still valid from `activeLeague`
- `PredictTab.tsx` — reads `session.userId`, `session.leagueId` — still valid
- `LeagueTab.tsx` — reads `session.leagueId`, `session.leagueName`, `session.userId`, `session.role` — still valid
- `RankingTab.tsx` — reads `session.leagueId` — still valid
- `ResultsTab.tsx` — reads `session.leagueId` — still valid
- `Join.tsx` — reads `session` to redirect if authenticated — behavior changes needed (see Pitfall 2)

**Minimal API contract to preserve:** The `Session` interface exported from `types.ts` must remain unchanged (`userId`, `nickname`, `leagueId`, `leagueName`, `role`). The refactor adds a parallel `leagues` array to `SessionContextValue` — components that don't need switching just keep consuming `session` as before.

**Example shape:**

### Pattern 2: Supabase RPC for `create_league`

**What:** A `SECURITY DEFINER` plpgsql function that atomically generates a unique invite code, inserts the league row, and inserts the creator as `role: 'admin'` in `league_members`. Returns `{ id, invite_code }`.

**Why RPC over Express endpoint:** The operation is a pure DB transaction — no external API calls, no complex business logic beyond string generation. An RPC keeps it atomic (a single round-trip, no partial state if the Express server times out mid-insert). The existing `lookup_league_by_invite_code` RPC proves this pattern works. [VERIFIED: SessionContext.tsx + Onboarding.tsx usage]

**Example plpgsql structure:**
```sql
-- Source: pattern extrapolated from existing lookup_league_by_invite_code [ASSUMED for exact SQL]
CREATE OR REPLACE FUNCTION create_league(p_name text, p_user_id uuid)
RETURNS TABLE(id uuid, invite_code text)
LANGUAGE plpgsql SECURITY DEFINER AS $$
DECLARE
  v_code text;
  v_id uuid;
  v_attempts int := 0;
BEGIN
  LOOP
    v_code := upper(substring(md5(random()::text) from 1 for 4));
    EXIT WHEN NOT EXISTS (SELECT 1 FROM leagues l WHERE l.invite_code = v_code);
    v_attempts := v_attempts + 1;
    IF v_attempts > 10 THEN RAISE EXCEPTION 'Could not generate unique code'; END IF;
  END LOOP;
  INSERT INTO leagues(name, invite_code, created_by) VALUES (p_name, v_code, p_user_id)
    RETURNING leagues.id INTO v_id;
  INSERT INTO league_members(league_id, user_id, role) VALUES (v_id, p_user_id, 'admin');
  RETURN QUERY SELECT v_id, v_code;
END;
$$;
```

**RLS / permissions note:** The RPC must be granted to the `authenticated` role (not `anon`) since the user must be signed in before creating a league. [ASSUMED: consistent with existing RPC grant patterns in this Supabase project — verify against live DB]

### Pattern 4: Countdown UX in PredictTab

**What:** For each OPEN match where `kickoff_at - now < 24h`, render a `"Locks in Xh Ym"` label below the date/time. Recalculate with `setInterval` at 60-second intervals so the countdown stays live.

**Where to insert:** In the per-match render block, between the date/time line and the bottom of the match card. Applies only when `match.uiStatus === "open"`.

**Time math:** [VERIFIED: PredictTab.tsx lines 73-86 — kickoff_at is a string from DB]
```typescript
// Source: derived from existing time calculation pattern in PredictTab.tsx
function getLocksInLabel(kickoffAt: string): string | null {
  const diff = new Date(kickoffAt).getTime() - Date.now();
  if (diff <= 0 || diff > 24 * 60 * 60 * 1000) return null;
  const h = Math.floor(diff / 3600000);
  const m = Math.floor((diff % 3600000) / 60000);
  return h > 0 ? `Locks in ${h}h ${m}m` : `Locks in ${m}m`;
}
```

**Interval strategy:** `useEffect` with a 60-second `setInterval` that increments a `tick` counter; the memoized `matches` array already recomputes on any dependency change. Adding `tick` as a dependency is the minimal-change approach. [ASSUMED: standard React countdown pattern — not verified against a specific doc]

### Anti-Patterns to Avoid

- **Don't introduce `single()` for membership query:** `.single()` throws a PGRST116 error when 0 rows match — this is what caused the original retry-loop problem. Use `.maybeSingle()` or array form.
- **Don't PUT active league in URL:** Sharing the current league context via URL state would complicate deep-linking; keep it in SessionContext (localStorage persistence is fine for MVP).
- **Don't fetch all leagues on every render:** Load leagues array once in `loadSession()`, not in each tab component. Cache with React Query if needed.
- **Don't skip the confirmation dialog before regenerating invite code:** D-05 explicitly requires it. Old deep-links break immediately — users must be warned.
- **Don't generate invite codes on the client:** Client-side generation bypasses the UNIQUE constraint retry loop. Always generate in the RPC. [VERIFIED: D-04, D-11]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Unique code collision avoidance | Custom retry loop in frontend | Supabase RPC with loop + DB UNIQUE constraint | DB is the only layer that can atomically check and insert; frontend has a race window |
| League switcher dropdown | Custom CSS dropdown | `shadcn/ui dropdown-menu.tsx` or `drawer.tsx` | Already installed; handles focus trap, keyboard nav, accessibility |
| Share invite link | Custom clipboard API wrapper | `navigator.share()` with `navigator.clipboard.writeText()` fallback | Already implemented in `LeagueTab.handleShare()` — reuse the exact same pattern |
| Countdown timer | `Date.now()` polling every frame | 60-second `setInterval` + tick state | Matches don't need sub-minute precision; 1-minute tick avoids unnecessary renders |
| Auth in create flow | New auth mechanism | `supabase.auth.signInAnonymously()` — same as join flow | Proven pattern from Phase 1; creator gets anonymous session, same as any joiner |

**Key insight:** The existing Supabase RPC pattern (`supabase.rpc(...)`) and the existing `shadcn/ui` component library eliminate the need for custom implementations across all new surfaces.

## Common Pitfalls

### Pitfall 1: Schema Divergence Between Migration File and Live DB

**What goes wrong:** The local migration file (`20260401000000_init_schema.sql`) does NOT contain `leagues`, `league_members`, `users`, or the RPC functions. These exist only in the live Supabase project. A developer writing a new migration must query the live DB types (in `packages/types/src/database.types.ts`) as the source of truth — not the migration file.

**Why it happens:** Phase 1 connected to a pre-existing Supabase project. The initial migration captured an old schema; the live schema diverged significantly.

**How to avoid:** Write a new migration file for Phase 2 changes (UNIQUE constraint on `invite_code`, new RPCs) using the live DB types as the baseline. Do not try to reconcile the old migration file. [VERIFIED: supabase/migrations/ — only one file, leagues table absent]

**Warning signs:** TypeScript types reference `leagues.invite_code` but the migration has no `leagues` table — this is expected and normal.

### Pitfall 2: Join.tsx Redirects Authenticated Users to `/`

**What goes wrong:** `Join.tsx` currently calls `navigate("/", { replace: true })` when `session` is truthy. After Phase 2, an authenticated user clicking a `/join/:code` link (to join a second league) gets silently redirected to the main app instead of being allowed to join.

**Why it happens:** Phase 1 assumed a user can only be in one league. [VERIFIED: Join.tsx lines 12-16]

**How to avoid:** In Phase 2, `Join.tsx` must detect the already-authenticated case and render an "Add this league?" confirmation UI instead of redirecting. The join logic calls the existing `INSERT league_members` flow without re-running auth.

**Warning signs:** Authenticated user visits `/join/ABCD`, gets bounced to `/` — the code in the URL is silently discarded.

### Pitfall 3: SessionContext `refreshSession()` Called from Onboarding Create Path

**What goes wrong:** After `create_league` RPC returns, the create path must call `refreshSession()` to load the new league into context. If `loadSession()` queries before the RPC transaction commits (unlikely but possible on high-latency connections), the session will load without the new league.

**Why it happens:** The RPC is `SECURITY DEFINER` and runs synchronously — by the time the client gets the response, the row IS committed. The race condition only applies if the client re-queries before the RPC responds (which it won't — the RPC response is awaited). [ASSUMED: Postgres transactional semantics; no Phase 1 equivalent observed]

**How to avoid:** Use the `{ id, invite_code }` returned by the RPC to pre-populate confirmation screen state WITHOUT waiting for `refreshSession()` to complete. Show the confirmation screen immediately; let `refreshSession()` load in the background. Enter app only after `refreshSession()` resolves.

**Warning signs:** Confirmation screen shows invite code correctly but tapping "Enter app" shows a blank screen — this is the race condition manifesting.

### Pitfall 4: Active League Persistence Across Page Reloads

**What goes wrong:** After implementing multi-league support, a user with two leagues always lands on the "first" league on every refresh because active league state lives only in React state (in-memory).

**Why it happens:** React state does not survive page reload.

**How to avoid:** Persist `activeLeagueId` in `localStorage`. On `loadSession()`, read `localStorage.getItem("activeLeagueId")` and cross-reference against the loaded memberships. If found, use it; otherwise default to the first membership. [ASSUMED: standard pattern for multi-context apps — not verified against Supabase docs]

**Warning signs:** User switches to league B, refreshes page, is back in league A.

### Pitfall 5: `invite_code` UNIQUE Constraint May Already Exist on Live DB

**What goes wrong:** Writing a migration that adds `UNIQUE(invite_code)` to `leagues` fails with "constraint already exists" if the live Supabase project already enforces it.

**Why it happens:** The live DB was built manually; constraints may or may not match what the migration file says.

**How to avoid:** Use `ALTER TABLE leagues ADD CONSTRAINT IF NOT EXISTS leagues_invite_code_key UNIQUE (invite_code)` or check the live DB via Supabase Dashboard before writing the migration. The migration should be idempotent. [VERIFIED: database.types.ts has no explicit constraint metadata — must be verified against live Supabase project schema view]
