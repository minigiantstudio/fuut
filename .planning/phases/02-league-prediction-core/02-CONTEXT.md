# Phase 2: League & Prediction Core - Context

**Gathered:** 2026-05-04
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 2 adds the ability to **create** leagues and makes the prediction flow complete end-to-end.

**Important:** Most of PREDICT-01 through PREDICT-04 shipped in Phase 1's parallel stream. Phase 2's real work is:
1. League creation flow (LEAGUE-01) — no create-league UI or API exists yet
2. Multi-league support with a league switcher (SessionContext refactor)
3. Prediction countdown UX (locking indicator enhancement)

**In scope:**
- Onboarding fork: "Create a league" vs "Join a league" on the first screen
- Create-league flow: Name league → Nickname → Email (mandatory for creator) → Confirmation screen → Enter app
- Auto-generated 4-char uppercase invite code, regeneratable by admin
- Multi-league membership: users can belong to multiple leagues
- TopBar league switcher (tap league name → dropdown/modal)
- "Locks in Xh Ym" countdown on OPEN matches when kickoff < 24h away

**Out of scope (Phase 4):**
- League admin tools: rename, remove members, reset scores (LEAGUE-03)
- League size limits / premium features (LEAGUE-04)

</domain>

<decisions>
## Implementation Decisions

### League Creation Flow

- **D-01:** First screen of `Onboarding.tsx` splits into two paths: **"Create a league"** and **"Join a league"**. This is a fork at step 1, not a separate route.
- **D-02:** Create flow step order: **Name league → Nickname → Email (mandatory) → Confirmation screen with code → Enter app**. Email is required for the creator (they become admin); optional for joiners (existing behavior unchanged).
- **D-03:** On confirmation screen, the invite code is shown **prominently with a share button** before the user enters the app. The code is also visible in the League tab's existing invite code card. Both surfaces.
- **D-04:** The invite code is **auto-generated** — a random 4-character uppercase code (e.g. `XKQZ`), generated by the backend at league creation time. Users cannot choose their own code.
- **D-05:** League admin can **regenerate the invite code** from the League tab (admin section). Old `/join/:oldcode` deep links stop working immediately when regenerated.

### Multi-League Support

- **D-06:** Users can belong to **multiple leagues simultaneously**. A user who creates one league can also join others.
- **D-07:** The **league switcher lives in the TopBar** — tap the league name to open a dropdown or bottom sheet showing all the user's leagues. Selecting a league makes it the active context for all tabs.
- **D-08:** `SessionContext` must be refactored from `.maybeSingle()` (one membership) to an array + active league state. All tabs read from the active league, not a hardcoded single membership.

### Prediction Locking UX

- **D-09:** When a match is in `OPEN` status and kickoff is less than 24 hours away, show a **"Locks in Xh Ym"** label beneath the kickoff date/time in `PredictTab`. Uses client-side time calculation from `match.kickoff_at`.
- **D-10:** Locking logic is unchanged — `kickoff_at <= now` triggers the LOCKED/NEEDS_RESULT state. The countdown is purely informational UI on top of the existing mechanism.

### Invite Code Format

- **D-11:** 4-character uppercase alphanumeric code, consistent with LEAGUE-02 requirement. Generation enforces uniqueness (retry on collision). The `leagues.invite_code` column should have a UNIQUE constraint if not already enforced.

### Claude's Discretion

- Implementation of the league creation Supabase function/RPC (generate + insert league row) — choose between a Supabase Edge Function, an RPC, or an Express API endpoint
- Exact UI for the TopBar switcher (dropdown vs. bottom sheet vs. modal)
- How the "Locks in" countdown re-calculates (re-render interval, or on-mount calculation only)
- Handling of the join flow when a user is already authenticated and hits `/join/:code` (join another league vs. error state)

</decisions>

- `SessionContext` — refactoring `loadSession()` is the most impactful change in this phase; all tabs depend on it

- `AppLayout` — passes `leagueName` down; may need to accept `onLeagueSwitch` callback
- Supabase `leagues` table — needs a `create_league` RPC or API endpoint; `created_by` FK to `users` is already in schema

<specifics>
## Specific Ideas

- Creator confirmation screen should show the invite code in the same large tracking-letter style used in the LeagueTab invite code card (`text-[14px] tracking-[0.3em]`)
- "Locks in Xh Ym" label should appear on the OPEN status rows only — not on SAVED rows (which have a prediction already)
- The admin regenerate button should show a confirmation dialog before invalidating the old code (friends with the old link would be locked out)

</specifics>
