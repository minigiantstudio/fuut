## Objetivo

Definir un flujo de trabajo claro para que:
- **el diseñador** prototipe en Lovable,
- **el arquitecto** traduzca el prototipo a una base técnica sólida,
- **los ingenieros** implementen con apoyo de agentes de IA,
- y todo el proceso mantenga control, calidad y trazabilidad.

## Resumen del flujo

### 1. Diseño
El diseñador crea un prototipo funcional en Lovable para validar:
- flujos de usuario,
- pantallas,
- estados vacíos / loading / error,
- navegación,
- intención visual.

**Salida:** prototipo + notas de handoff.

### 2. Arquitectura
El arquitecto toma el prototipo y define:
- estructura del repositorio,
- modelo de dominio,
- contratos de API,
- documentación,
- definición de agentes,
- reglas y límites.

**Salida:** blueprint técnico + documentación base + tareas preparadas.

### 3. Implementación
El ingeniero toma tareas pequeñas y usa un agente para ayudar a implementarlas.

**Regla clave:** el agente no recibe “haz toda la app”, sino tareas concretas con:
- objetivo,
- alcance,
- archivos permitidos,
- restricciones,
- criterios de aceptación,
- validación.

### 4. Revisión
Antes de merge:
- revisar diff,
- correr tests,
- validar arquitectura,
- revisar seguridad,
- actualizar documentación.

***

## Principios clave

1. **Lovable sirve para prototipar, no para definir la producción.**
2. **El arquitecto define el sistema, no solo la app.**
3. **El ingeniero usa IA como acelerador, no como reemplazo del criterio técnico.**
4. **Toda tarea debe ser pequeña, clara y verificable.**
5. **Prompts, políticas y tareas deben vivir en el repo.**
6. **Cambios sensibles requieren revisión humana.**

***

## Estructura recomendada del repo

- `docs/product/`
- `docs/architecture/`
- `docs/engineering/`
- `docs/agents/`
- `.ai/prompts/`
- `.ai/policies/`
- `.ai/tasks/`
- `src/`
- `infra/`

***

## Entregables por rol

### Diseñador
- Prototipo funcional.
- Lista de pantallas.
- Notas UX.
- Edge cases.
- Decisiones que deben conservarse.

### Arquitecto
- System overview.
- Domain model.
- Capability matrix.
- ADRs.
- Definición de agentes.
- Task breakdown.

### Ingeniero
- Código productivo.
- Tests.
- PRs.
- Evidencia de validación.
- Correcciones según revisión.

***

## Diagrama mental del proceso

**Idea → Prototipo en Lovable → Blueprint técnico → Tareas pequeñas → Implementación con agente → Revisión humana → Tests → Merge**

***

4. `definition-of-done.md`

Eso te da un flujo mínimo pero serio, fácil de adoptar por el equipo.
